## Crate shape — standalone packages

Both `mock/crates/mockspace-template/` and `mock/crates/mockspace-config/` are **standalone Cargo packages**, not workspace members of mockspace's root `Cargo.toml`.

Rationale (per lifecycle decision D6 in the topic file):
- Mock-crates exist for design validation, not for the mockspace binary to consume.
- The mockspace bin keeps using its internal `src/config.rs`, `src/render_*.rs`. Live code stays independent.
- Homma path-deps each mock-crate directly: `mockspace-template = { path = "../mockspace/mock/crates/mockspace-template" }`.
- Standalone packages compile independently via `cd mock/crates/<name> && cargo build`. Each has its own `Cargo.lock` (gitignored).

`mock/Cargo.toml` (a sub-workspace) does NOT exist this round; #224 (mockspace self-host) introduces it.

`.gitignore` adjustments: `mock/crates/*/Cargo.lock` and `mock/crates/*/target/` get added so standalone-package build output stays out of git. Done in this round.

`Template` + `TemplateEnv` wrapping minijinja.

```rust
use minijinja::{Environment, UndefinedBehavior};
use serde::Serialize;
use crate::error::RenderError;

pub struct TemplateEnv {
    inner: Environment<'static>,
}

impl TemplateEnv {
    pub fn new() -> Self {
        let mut inner = Environment::new();
        inner.set_undefined_behavior(UndefinedBehavior::Strict);
        // autoescape off by default for non-HTML templates
        inner.set_auto_escape_callback(|_| minijinja::AutoEscape::None);
        Self { inner }
    }

    pub fn add_template(&mut self, name: &str, source: &str) -> Result<(), RenderError> {
        self.inner.add_template_owned(name.to_string(), source.to_string())?;
        Ok(())
    }

    pub fn get_template<'a>(&'a self, name: &str) -> Result<Template<'a>, RenderError> {
        let t = self.inner.get_template(name)
            .map_err(|_| RenderError::TemplateNotFound(name.to_string()))?;
        Ok(Template { inner: t })
    }

    pub fn render_str<C: Serialize>(&self, source: &str, ctx: &C) -> Result<String, RenderError> {
        Ok(self.inner.render_str(source, ctx)?)
    }
}

impl Default for TemplateEnv {
    fn default() -> Self { Self::new() }
}

pub struct Template<'env> {
    inner: minijinja::Template<'env, 'env>,
}

impl<'env> Template<'env> {
    pub fn render<C: Serialize>(&self, ctx: &C) -> Result<String, RenderError> {
        Ok(self.inner.render(ctx)?)
    }
}
```

## Deviations from doc CL

Recorded honestly during IMPL:

### LintConfig re-export dropped

**Doc CL section 4.2 / 4.6** said `mockspace-config` re-exports `LintConfig` from `mockspace-lint-rules`. **Actual implementation:** `Config.lint_overrides` is `BTreeMap<String, String>` (lint-name → severity-string). Consumers parse the strings into typed `Severity` via `mockspace-lint-rules` at usage time, not at config-parse time.

**Why:** Re-exporting LintConfig cross-crate would create a hard dep on `mockspace-lint-rules` for any tool that just wants to read the schema. Decoupled string-keyed map keeps the schema independent of the lint-rule crate's evolution.

**Impact:** Documented in `BACKLOG.md.tmpl` under "Validation extensions"; consumers wanting typed severities go through mockspace-lint-rules directly.
