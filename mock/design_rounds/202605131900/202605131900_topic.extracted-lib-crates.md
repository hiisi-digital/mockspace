# Topic: extract reusable internals as lib crates

**Date:** 2026-05-13
**Phase:** TOPIC
**Scope:** mockspace internals → reusable lib crates that homma will consume
**Related tasks:** #444 (this round), #224 (self-host mockspace, separate), #445 (homma bootstrap, downstream consumer)

## Framing

The clause-dev workspace is gaining a new tool, `homma`, that needs to consume parts of mockspace's internals: template engine, config primitives, and agent-file generation. Today those live as private modules inside mockspace's binary crate. This round designs the extraction into reusable lib crates so homma can depend on them without depending on the mockspace binary or its mockspace-specific machinery (design rounds, bootstrap, bench harness, dylib check, etc.).

The extraction lives in `mock/crates/<name>/` (design-only). Live root code (`src/*.rs`) is not touched in this round. A later round ports the live code to consume the validated mock-crate shapes.

### What homma actually consumes

Three load-bearing things:

1. **Template engine**: Mustache-like minimal renderer for `homma.toml` templated values (`{{owner}}`, `{{repo}}`, `{{host}}` etc.) and for workspace-level agent-rule templates.
2. **Config primitives**: TOML schema base shared between `mockspace.toml` and `homma.toml`, so both tools speak compatible config dialects.
3. **Agent-file rendering**: homma workspaces, like mockspace consumer repos, generate fresh `.claude/` and `.github/copilot/` from templates each run (gitignored, never committed). Homma needs the same engine.

(Lint runner extraction dropped from this round. Viola integration will replace mockspace's current lint runner; revisit then. No workspace-level lints needed out of the gate.)

## Decisions

### D1: Crate granularity

**Decided:** Two reusable lib crates.

- `mockspace-template` — template engine + agent-file rendering (rendering is a specialised consumer of the template engine; they live together)
- `mockspace-config` — config primitives (TOML schema base, model types)

Rationale:
- Templating and agent rendering share enough internals that splitting them adds cost without benefit. Agent rendering IS templating-with-platform-specific-output-paths.
- Config is independent; templating doesn't fundamentally require typed config (substitutions can come from anywhere).
- Lint runner stays inside mockspace for now; viola will replace it.
- Names: `mockspace-config` (not `-config-base`) — simpler, the "base" suffix would only make sense if there were multiple config crates.

### D2: Template engine power

**Decided:** Full template language via `minijinja`.

Why minijinja over alternatives:
- Lighter than `tera` (~100KB vs ~300KB compiled), faster compile times.
- Jinja2-syntax clone by Armin Ronacher (Jinja's original author); battle-tested semantics.
- `no_std`-friendlier than tera.
- Provides loops (`{% for %}`), conditionals (`{% if %}`), includes (`{% include %}`), filters — future-proof for richer rendering (e.g. "emit a rule fragment per crate", workspace-level docs generation).
- Uses `serde::Serialize` for template context, which decouples cleanly from config typing.

Pin chosen at lock time; revisit if minijinja becomes unsuitable.

### D3: Inter-crate dependency direction

**Decided:** `mockspace-template` and `mockspace-config` are independent (no inter-crate dep).

- `mockspace-template` wraps `minijinja`. Template context is any `T: serde::Serialize`. No knowledge of `Config`.
- `mockspace-config` is standalone TOML schema + model types. No knowledge of templating.
- Consumers (mockspace tool, homma) wire them: parse config → serialize to template context → render templates.

This means homma can use `mockspace-template` even if it doesn't use `mockspace-config` (e.g. if homma evolves a wholly different config schema). It also means mockspace-config can be consumed independently by tools that care about TOML schema only, not templates.

### D4: mockspace-config scope — canonical Config + `IntoMockspaceConfig` trait

**Decided:** `mockspace-config` exposes:

1. **The canonical typed `Config` struct** — the template-context contract every mockspace template renders against. Field names (`mock_dir`, `crates_dir`, `project_name`, `crate_prefix`, etc.) are mockspace's stable vocabulary that templates hard-code references to.
2. **TOML parser for mockspace.toml** — the default entry point producing `Config`.
3. **An `IntoMockspaceConfig` trait** — consumer apps with their own config schemas (homma.toml has hosts/orgs/repos; future tools may have their own shapes) implement this trait on their typed config to produce a valid `Config`. The trait makes intent explicit and discoverable; type-safe; compile-time-checked.

Sketch:
```rust
pub trait IntoMockspaceConfig {
    fn into_mockspace_config(self) -> Result<Config, MappingError>;
}

// Default impl: mockspace.toml's own parsed shape IS a Config (identity).
// Consumers like homma write:
impl IntoMockspaceConfig for HommaConfig {
    fn into_mockspace_config(self) -> Result<Config, MappingError> {
        Ok(Config {
            mock_dir: self.workspace.root.join("mock"),
            project_name: self.workspace.name,
            // ...etc, mapping homma's fields onto mockspace's vocabulary
        })
    }
}
```

Rationale:
- Templates use mockspace's field names as the lingua franca. Any consumer that wants to render those templates produces a `Config` first.
- Trait-based (chosen over declarative path-rewrite) because the mapping is naturally compile-time-checked; consumer schemas evolve in lockstep with their typed config in Rust; no data-driven mapping language to maintain.
- mockspace.toml stays the canonical reference schema; other configs are translations to that vocabulary.
- This makes `mockspace-config` genuinely shared infrastructure: any tool can consume it, and templates work uniformly across config schemas as long as a translation exists.

### D5: Agent-render Platform extensibility

**Decided:** Trait-based. `mockspace-template` defines a `Platform` trait. Built-in impls: `ClaudePlatform`, `CopilotPlatform`. External consumers can add new platforms (`CursorPlatform` etc.) by implementing the trait.

Detailed trait surface TBD in the doc CL.

### D6: Lifecycle — design + implementation in mock/crates/; live-code port deferred

**Decided:** This round writes both doc CL (per-crate `DESIGN.md.tmpl`) AND src CL (mock-crate `src/*.rs` implementation). Both crates live under `mock/crates/`. The mockspace binary at root is NOT touched this round (its internal `src/config.rs`, `src/render_*.rs` etc. stay as-is).

Lifecycle:
1. **This round (#444):** design + implement extracted crates in `mock/crates/mockspace-template/` and `mock/crates/mockspace-config/`. Both have full source + design docs.
2. **homma bootstrap (#445):** homma consumes the mock-crate shapes via explicit path-dep (`{ path = "../mockspace/mock/crates/mockspace-template" }`). This validates the extraction by exercising the API against a real second consumer.
3. **Future round (post-validation):** once homma proves the shape works and any revisions stabilize, port the validated `mock/crates/<name>/src/` code into the live root replacing the internal modules; publish to crates.io. mockspace bin starts consuming via crates.io versions.

Rationale:
- An unimplemented design template cannot be validated; mockspace-template/config need real code for homma to exercise.
- Keeping the live mockspace bin untouched preserves continuity: existing mockspace consumers (arvo, hilavitkutin, etc.) keep working unchanged through the extraction-validation period.
- The path-dep is intentionally awkward (`../mockspace/mock/crates/...`) so it doesn't look like a published consumer relationship and isn't accidentally relied on long-term.
