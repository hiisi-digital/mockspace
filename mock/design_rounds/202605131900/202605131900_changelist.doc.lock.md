
### 1. `mock/crates/mockspace-template/DESIGN.md.tmpl`

**Purpose:** primary design document for the template+agent-render extraction.

**Sections (in order):**

#### 1.1 `# mockspace-template`

Top-line title + one-paragraph framing: this crate is the reusable template engine and agent-file generator extracted from mockspace's `src/render_*.rs` modules. Consumers include the mockspace binary, homma, and any future tool that needs to render Jinja2-style templates against typed contexts and emit agent integration files to platform-specific destinations.

#### 1.2 `## What this crate provides`

Bulleted list of the public surface at a high level:

- `Template` — a single compiled template (wraps `minijinja::Template<'_, '_>`); rendered against a `serde::Serialize` context.
- `TemplateEnv` — a registry of named templates with shared filters/functions; wraps `minijinja::Environment<'_>`.
- `Platform` trait — abstract over agent platform specifics (output paths, frontmatter format, hook-helper injection).
- `ClaudePlatform` — built-in `impl Platform` for the Anthropic Claude Code surface (`.claude/rules/`, `paths:` frontmatter, hookSpecificOutput JSON).
- `CopilotPlatform` — built-in `impl Platform` for the GitHub Copilot CLI surface (`.github/instructions/`, `applyTo:` frontmatter, flat JSON decisions).
- `AgentRenderer` — walks a source template directory, renders each `.md.tmpl` file against a context, and emits per-platform output trees.
- `walk_template_tree(src, dst, env, ctx)` — generic recursive renderer for non-agent template trees.

#### 1.3 `## Template engine — minijinja wrapper`

Describe the Template/TemplateEnv layer:
- Why minijinja: lighter than tera (~100KB compiled), Jinja2-syntax clone by Armin Ronacher, fast compile, `no_std`-friendlier, `serde::Serialize` context model.
- How a Template is built: `TemplateEnv::add_template(name, source)?` then `env.get_template(name)?.render(ctx)?`.
- Context model: any `T: serde::Serialize`. Mockspace passes its `Config` (after `IntoMockspaceConfig`); homma passes its rendered `Config` (after its own `impl IntoMockspaceConfig`).
- Autoescape policy: **off** by default. The templates emit Markdown/Rust/shell text, never HTML.
- Strict-mode setting: **on**. Undefined template variables produce errors at render time, not silent empty-string substitution.
- Custom filter / function registrations: empty for v1; future BACKLOG.

#### 1.4 `## Platform trait`

Describe the trait that abstracts agent-platform specifics. Sketch:

```rust
pub trait Platform {
    /// Identifier used in templates and CLI output (e.g. "claude", "copilot").
    fn name(&self) -> &'static str;

    /// Translate a source template's logical name to the platform's output path.
    /// e.g. ClaudePlatform: "rules/foo" -> ".claude/rules/foo.md"
    ///      CopilotPlatform: "rules/foo" -> ".github/instructions/foo.instructions.md"
    fn output_path(&self, repo_root: &Path, logical_name: &str) -> PathBuf;

    /// Generate the platform-specific frontmatter for a rule's logical paths.
    /// Claude uses `paths:`, Copilot uses `applyTo:`.
    fn frontmatter(&self, applies_to: &[&str]) -> String;

    /// Generate the platform-specific hook helper snippet to inject at
    /// {{HOOK_HELPERS}} substitution points. Different between Claude
    /// (hookSpecificOutput JSON) and Copilot (flat JSON / plain text).
    fn hook_helpers(&self, repo_root: &Path) -> String;

    /// Platform-specific config file (e.g. settings.json) emission.
    /// Returns Maybe<(path, contents)>; None if the platform has no such file.
    fn settings_file(&self, repo_root: &Path, hooks: &[HookDecl]) -> Option<(PathBuf, String)>;
}
```

Note: `Maybe<T>`/`Outcome<T,E>` not used here — this crate is in mockspace's repo where notko/arvo are not workspace deps. Document this exception explicitly: mockspace-template uses std `Option`/`Result`/`PathBuf`/`String` because mockspace is the meta-tool and the workspace-substrate-discipline rules apply to consumers of the substrate, not to the design-tool itself. (When mockspace-template eventually becomes a consumer of arvo+notko via the live-port round, this gets revisited.)

#### 1.5 `## Built-in Platform impls — ClaudePlatform / CopilotPlatform`

Per-platform tables describing exact path mapping, frontmatter shapes, hook helper bodies. The bodies port verbatim from `src/render_agent.rs` `CLAUDE_HOOK_HELPERS` and `COPILOT_HOOK_HELPERS` const strs.

Output path mapping:

| Logical | ClaudePlatform output | CopilotPlatform output |
|---|---|---|
| `rules/<name>` | `.claude/rules/<name>.md` | `.github/instructions/<name>.instructions.md` |
| `skills/<name>` | `.claude/skills/<name>/SKILL.md` | `.github/skills/<name>/SKILL.md` |
| `hooks/<name>` | `.claude/hooks/<name>.sh` | `.github/hooks/<name>.sh` |
| `instructions` (root) | `.claude/CLAUDE.md` | `.github/copilot-instructions.md` |

Frontmatter:

| Field | ClaudePlatform | CopilotPlatform |
|---|---|---|
| Applicability glob | `paths: ["**/*.rs", ...]` | `applyTo: "**/*.rs,..."` |

Hook helpers: each platform's helper body is platform-private (Claude uses `hookSpecificOutput` JSON wrapper; Copilot uses flat `{"permissionDecision":...}` JSON; different `_extract` shapes for input field access).

#### 1.6 `## AgentRenderer — orchestration`

Describe the directory-walking renderer:

```rust
pub struct AgentRenderer<'env, P: Platform> {
    env: &'env TemplateEnv,
    platform: P,
    src_root: PathBuf,
    dst_root: PathBuf,
}

impl<'env, P: Platform> AgentRenderer<'env, P> {
    pub fn new(env: &'env TemplateEnv, platform: P, src: PathBuf, dst: PathBuf) -> Self;
    pub fn render_all(&self, ctx: &impl serde::Serialize) -> Result<RenderReport, RenderError>;
}
```

Walks `src_root` for `*.md.tmpl` files. For each, derives logical name (path relative to src_root with `.md.tmpl` stripped), looks up platform output path via `platform.output_path()`, renders the template through `env`, writes to dst path. Returns a report listing every file written.

#### 1.7 `## Non-agent template rendering — walk_template_tree`

Free function for non-agent template trees (e.g. workspace-docs generation):

```rust
pub fn walk_template_tree(
    env: &TemplateEnv,
    src_root: &Path,
    dst_root: &Path,
    ctx: &impl serde::Serialize,
) -> Result<RenderReport, RenderError>;
```

Same shape as AgentRenderer but no platform-specific path mapping; preserves source structure with `.tmpl` stripped from filenames.

### 4. `mock/crates/mockspace-config/DESIGN.md.tmpl`

**Sections:**

#### 4.1 `# mockspace-config`

Top-line + framing: the canonical typed schema for mockspace.toml plus the `IntoMockspaceConfig` trait so consumer schemas (homma.toml etc.) can map onto mockspace's template-context vocabulary. The crate is the abstraction over "where Config values come from"; mockspace.toml is the default source.

#### 4.2 `## What this crate provides`

- `Config` — the canonical typed struct (verbatim port of `mockspace/src/config.rs:37` minus mockspace-internal-only fields).
- `parse_mockspace_toml(path: &Path) -> Result<Config, ConfigError>` — default entry point for mockspace.toml consumers.
- `IntoMockspaceConfig` trait — consumers with their own typed config produce a Config via this trait. Required for any tool that wants to render mockspace templates against a non-mockspace.toml config source.
- `ConfigError` — error type covering parse failures, missing required fields, mapping errors.
- Sub-types: `InstallMode`, `AttributionConfig`, `MacroStyle`, `CommitStyle`, `LintConfig` re-export.

#### 4.3 `## The canonical Config schema`

Field-by-field table reproducing the current `Config` struct surface. Brief description per field. This is the contract templates rely on.

| Field | Type | Purpose |
|---|---|---|
| `mock_dir` | `PathBuf` | Path to the mockspace workspace root (typically `<repo>/mock`). |
| `crates_dir` | `PathBuf` | Path to the mock workspace's crates directory. |
| `repo_root` | `PathBuf` | Path to the consumer repo root. |
| `docs_dir` | `PathBuf` | Path where rendered docs go. |
| `project_name` | `String` | Display name for the project. |
| `crate_prefix` | `String` | Common prefix for crates (e.g. "arvo-"). |
| `proc_macro_crates` | `Vec<String>` | Names of proc-macro crates (lint exemption). |
| `lint_proc_macro_source` | `bool` | Whether to lint proc-macro source code. |
| `module_crates` | `Vec<String>` | Crates published as separate modules. |
| `unprefixed_crates` | `Vec<String>` | Crates that don't get the prefix. |

Mark which fields are template-context-visible (most) vs purely behavioral (install_* modes, abi_version, etc.).

#### 4.4 `## IntoMockspaceConfig trait`

Full trait definition with the sketch from D4. Document:
- The trait is auto-implemented for `Config` itself (`impl IntoMockspaceConfig for Config { ... self }`).
- Consumer schemas write the trait impl mapping their fields onto Config fields.
- `MappingError` for "this consumer config doesn't have enough info to produce a valid Config" cases (e.g. mock_dir not in homma.toml).
- Templates consume the `serde::Serialize` view of `Config`, NOT the raw consumer config, so the trait is the bridge.

Example impl block shown verbatim for homma (as a forward-reference; homma will implement it for real later).
